Welcome!

Web application development is the process of creating a web site that acts like an application program. Web applications collect, manipulate, calculate and display data, using the same programming constructs as apps you download and install. Web apps do not need to be installed. The code is run within your browser, using data collected on demand from web services.

Modern web applications use the following key technologies and organizing frameworks:

  • JavaScript and sophisticated JavaScript libraries, such as React and AngularJS, to create dynamic data-driven web pages
  • Single-page web apps that behave like standalone applications but do not require downloading and installation
  • Asynchronous function calls to communicate with web services directly from a web page
  • JSON (JavaScript Object Notation) to encode complex data in a portable compact human-readable format

Background

Every web browser knows how to run JavaScript code. With JavaScript, you can make a web page into an interactive web application, like a game or an interactive data dashboard. With JavaScript you can

  • make parts of web pages appear, disappear, and move around
  • respond to mouse clicks, finger taps, drag and drop, and keyboard entry
  • calculate complex mathematical formulas
  • repeat operations in loops that can run a hundreds or thousands of times

Basically, with JavaScript in a browser, you can do anything you can do in any other programming language except read or write files on your computer. It is prevented from doing that so that hackers can't write web pages that steal information from you or damage your computer files.

2. Command shells

A command shell or terminal window is a program that lets you run commands and manage files using text instructions rather than point and click. Decades ago, all interactions with a computer were done using command shells. Graphical user interfaces like MacOS and Windows replaced command shells for everyday users, but programmers still use the command line for much development. One reason is that there are some tasks that can only be done with a command line. Another is that it is much easier to write scripts to do multi-step tasks with command lines than to write such scripts for graphical user interfaces.

Working with command shells is very different than point and click. With practice it will become intuitive, but when you start, you will likely make frequent mistakes, putting files in the wrong place, running commands with the wrong parameters. Don't worry. We all did that.

3. Git and Github

Git is a desktop application that you install and use to keep track of changes you make to a code project. The git term for a collection of files is repository or, informally, repo. Typically you will make a new repository for every code project. 

When you are happy with some changes you have made to your code, you use the git commit command to save these changes in a local history with a comment about why the changes were made, e.g., "fixed zip code bug". This is called a commit. This can be done in the command shell or in a good editor, like VS Code.

Github is a web site that lets you store a remote copy of your local repository, including the history of commits. Other web sites that do this are Bitbucket and Gitlab. With all of them you use

  • git clone to download a local copy of a remote repository
  • git push to upload changes committed in the local repository to the remote version.
  • git pull to download changes in the remote version to the local copy 

A team of developers can work together on a remote repository. Each developer pulls the current version, makes changes to the local copy, tests, commits, and pushes the changes to the remote repository for other developers to access. The commit history on the remote repository helps developers keep track of what the team has done.

Background

API Keys

Web services that return data in JSON form have become very popular. They allow a data consumer web site A to get data from a data producer web site B, manipulate that data, and display it to end users. One concern for the producer is that some consumers may overwhelm the producer’s servers with frequent requests for large amounts of data.

To control this, many data producers require that all requests include an API key. This is a long sequence of characters that you need to include in every request for data. You can often get an API key for free. The producer uses the key to track who is asking for data. The producer can easily limit how often you can ask for data and how much data you can get in one call. The producer may allow a consumer to pay for increased access.

JSON

JSON (JavaScript Object Notation) is a format for representing complex data in plain text form. As the name suggests, the format originated with JavaScript. Douglas Crockford proposed using it as a general data exchange format instead of the more verbose XML (Extensible Markup Language). While XML was very general, powerful and popular, JSON quickly became more popular, because it was easier to write code for and was comparatively more compact.

Web pages: dynamic, interactive, single-page

A web page is dynamic if the HTML is modified at run-time by JavaScript based on some data. For example, in the application for this task, JavaScript will fetch data from a data producer, then modify an HTML table to show that data. The browser displays the HTML the same as it displays HTML you write by hand.

A dynamic web page is interactive if it changes based on user actions, such as selecting an item from a menu or entering some text.

Bootstrap

Bootstrap is a very popular collection of CSS classes and JavaScript for styling web pages, with special emphasis on mobile devices. Bootstrap CSS classes can be used directly in the JSX-generated HTML, e.g., 

<table class="table table-striped table-bordered">
...
</table>

Bootstrap includes classes for complicated compound objects like navigation bars, with links and dropdown menus. Bootstrap also has a large number of CSS classes for doing responsive layout - that is, layouts that don't just shrink but can rearrange columns based on screen size.

Your Task

You have been asked to create two web pages with statistics on the Senate and the House. This will involve:

  • looping over the data about the members of the two houses, counting and adding as you go
  • constructing an HTML table to display the final results with clear labels and properly formatted numbers

Each statistic requires different logic to calculate. Some are simple counts, others are complex. The best way to organize such code is to define a function to calculate each statistic. Each function may in turn call other functions.

Background

Models and Views

Best practice suggests separating the calculation of the statistics from the construction of the HTML. This is called separating the model from the view. That way your calculation code can be written and tested separately from the HTML code. Also, the calculation code could more easily be reused in future programs.

The steps in this Plan of Attack recommend creating a JSON object to hold all the statistics. This is the model. The JSX to display the statistics will be the view. There should be as little JavaScript code in the JSX as possible. 

Vertical Slices

There are several ways to approach developing screens with complex models:

  • Front end first: Develop the JSX for all the tables on both screens first, then add the code to calculate the statistics.
  • Back end first: Write the code to calculate all the statistics, then develop the JSX to display them.

Agile recommends neither. Instead, develop in vertical slices. Do one piece of end-to-end testable functionality at a time, i.e., calculate and display one statistic, then the next, then the next, until finished. That way you learn what is the best way to organize the calculated results for display and work out any formatting issues as early as possible.

For this task, one way to do that it to create the initial pages, with the static text but not tables.  Then add the different statistics to the pages, one by one.

Technical Debt and Refactoring

Technical debt is a term in agile to describe the accumulation of hard to maintain code.  You incur technical debt when you write

  • very similar code repeated in multiple places
    • This means a programmer has to make fixes and improvement in multiple places.
  • very long functions
    • This makes the function harder for a programmer to read, understand, reuse elsewhere. 
  • vague or misleading names for variables and functions
    • This confuses a programmer trying to debug or extend your code.

Refactoring is a programming term for cleaning code up. It means changing the code to make it more maintainable. Refactoring should never change what a code does, just how maintainable it is.

Because this task involves defining a number of similar functions, it is ripe for incurring technical debt, and is a good opportunity to practice refactoring.

Put real data in the Senate file

You will need to define code to fetch JSON and generate HTML. Most of that code should be in general reusable functions defined in a separate JavaScript file, say scripts/html.js.

The Senate HTML file should call those functions, passing information specific to the file, such as what data file to use and where to put the HTML. Something like this:

<script type="module">
  import { fetchJson, makeHtml } from './scripts/html.js';

  const data = fetchJson(...);
  document.querySelector(...).innerHTML = makeMemberRows(data);
</script>

Making this script element a module enables the use of import to load functions from an external JavaScript file.

Unfortunately, this won't quite work!  A function like fetchJson can only return a promise of data, not actual data. Since await in a top-level module is not yet supported in some browsers, like Safari, the portable way to fetch data uses a then callback on the promise returned by the fetch.

<script type="module">
  import { fetchJson, makeHtml } from './scripts/html.js';

  fetchJson(...).then((data) => {
    document.querySelector(...).innerHTML = makeMemberRows(data);
  });
</script>

This will start fetching the data, but not freeze the browser until a response arrives. When the data does finally arrive, the function inside the then() will be called to take the data, make the HTML, and store it on the page. 

Use this approach to define a script element in index.html to fetch the Senate data from the local file. When that data arrives, the script should then call a function to generate the table rows for the members of the Senate.

You will have to fill in the ... parts in the above fragment with (1) the URL for your Senate JSON data file, and (2) the CSS selector for the body of your Senate table.
